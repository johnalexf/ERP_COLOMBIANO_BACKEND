# MANUAL TÉCNICO: COMPLEMENTO DE SEGURIDAD Y AISLAMIENTO MULTI-TENANT (DOMINIO INQUILINOS)

## 1. OBJETIVO Y CONTEXTO ARQUITECTÓNICO
El propósito de este componente es resolver el conflicto de colisión de identificadores (`user_id`) derivado del uso de bases de datos independientes para el entorno administrativo (Admin) y el entorno de inquilinos (Tenant). 

Para garantizar el aislamiento estricto, se implementa una estrategia de "Defensa en Profundidad" dividida en tres capas criptográficas y operativas:
1. **Inyección de Claims Personalizados:** Marcado del token JWT con el origen del ciclo de vida del usuario.
2. **Sobreescritura del Motor de Autenticación:** Desvío de la resolución global de usuarios hacia consultas exclusivas en la tabla de inquilinos.
3. **Validación Per-View de Permisos:** Control de acceso basado en políticas para interceptar peticiones cruzadas maliciosas.

---

## 2. CAPA 1: INYECCIÓN DE METADATOS EN EL PAYLOAD (VIEWS.PY)
Se modifica la vista encargada de la emisión de credenciales (`TenantLoginView`) para añadir una reclamación personalizada (`claim`) dentro del cuerpo no encriptado pero firmado del JSON Web Token.

### Implementación del Endpoint de Autenticación:
```python
from rest_framework.views import APIView
from rest_framework.response import Response
from rest_framework import status
from rest_framework_simplejwt.tokens import RefreshToken
from .serializers import TenantLoginSerializer

class TenantLoginView(APIView):
    """
    Procesa la autenticación de inquilinos y emite tokens estructurados.
    """
    serializer_class = TenantLoginSerializer

    def post(self, request):
        serializer = self.serializer_class(data=request.data)
        serializer.is_valid(raise_exception=True)
        user = serializer.validated_data['user']
        
        # Instanciación del token de refresco (Token Padre)
        refresh = RefreshToken.for_user(user)
        
        # INYECCIÓN DEL CLAIM CRIPTOGRÁFICO
        # Se añade el metadato de origen. SimpleJWT hereda automáticamente
        # este parámetro al Access Token (Token Hijo).
        refresh['user_type'] = 'tenant'
        
        return Response({
            'refresh': str(refresh),
            'access': str(refresh.access_token),
            'message': 'Autenticación exitosa.'
        }, status=status.HTTP_200_OK)
```

## 3. CAPA 2: MOTOR DE AUTENTICACIÓN PERSONALIZADO (AUTHENTICATION.PY)
Se implementa un mecanismo de resolución de identidad específico para el entorno inquilino. Este componente intercepta el token antes de la evaluación de la vista y altera el comportamiento nativo de Django REST Framework, obligando al sistema a interactuar con el modelo de datos correcto.

### Creación del archivo `apps/tenant/tenant_users/authentication.py`:
```python
from rest_framework_simplejwt.authentication import JWTAuthentication
from rest_framework_simplejwt.exceptions import AuthenticationFailed
from .models import TenantUser

class TenantJWTAuthentication(JWTAuthentication):
    """
    Clase de autenticación especializada para mitigar colisiones de ID.
    Reemplaza la búsqueda en la tabla global por una consulta dirigida a TenantUser.
    """
    def get_user(self, validated_token):
        # Extracción de la carga útil del token verificado
        user_id = validated_token.get('user_id')
        user_type = validated_token.get('user_type')

        # Principio Fail-Fast: Rechazo inmediato si el token pertenece a otra aplicación
        if user_type != 'tenant':
            raise AuthenticationFailed(
                'Token inválido para este entorno operativo.', 
                code='invalid_token'
            )

        # Resolución dirigida de la entidad en la base de datos correspondiente
        try:
            user = TenantUser.objects.get(id=user_id)
        except TenantUser.DoesNotExist:
            raise AuthenticationFailed(
                'Usuario inquilino no encontrado en los registros.', 
                code='user_not_found'
            )

        # Verificación del estado operativo de la cuenta
        if not user.is_active:
            raise AuthenticationFailed(
                'Esta cuenta de inquilino se encuentra inactiva.', 
                code='user_inactive'
            )

        # Hidratación exitosa del objeto request.user
        return user
```

## 4. CAPA 3: DIRECTIVA DE PERMISOS DE ACCESO CONTROLADO (PERMISSIONS.PY)
Mecanismo de control de acceso basado en el estado del token descentralizado. Funciona de manera independiente a la base de datos para validar las reglas de negocio de la ruta.

### Creación del archivo `apps/tenant/tenant_users/permissions.py`:
```python
from rest_framework.permissions import BasePermission

class IsTenantUser(BasePermission):
    """
    Garantiza que el portador del token cuenta con los privilegios de acceso
    específicos del dominio de inquilinos mediante la lectura del claim 'user_type'.
    """
    def has_permission(self, request, view):
        # Evaluación de la existencia de una firma válida descodificada
        if request.auth:
            user_type = request.auth.get('user_type', None)
            
            # El acceso es concedido únicamente ante la coincidencia exacta
            return user_type == 'tenant'
            
        return False
```

## 5. INTEGRACIÓN Y CONFIGURACIÓN DE LA CAPA DE PRESENTACIÓN (VIEWS.PY)
Para activar la arquitectura de seguridad diseñada, se realiza un *override* de las propiedades heredadas en el controlador maestro (`ModelViewSet`). Esto desactiva el comportamiento configurado en el `settings.py` general únicamente para este conjunto de endpoints.

### Modificación del Controlador de Usuarios Tenant:
```python
from rest_framework import viewsets
from rest_framework.permissions import AllowAny, IsAuthenticated
from django.conf import settings
from .models import TenantUser
from .serializers import TenantUserSerializer

# Importación de los componentes de seguridad personalizados
from .authentication import TenantJWTAuthentication
from .permissions import IsTenantUser

class TenantUserViewSet(viewsets.ModelViewSet):
    queryset = TenantUser.objects.all()
    serializer_class = TenantUserSerializer
    
    # ASIGNACIÓN DEL MOTOR LOCAL (Reemplaza la política global de DRF)
    authentication_classes = [TenantJWTAuthentication]

    def get_permissions(self):
        # Apertura de canales para registros públicos o entornos locales controlados
        if settings.DEBUG or self.action == 'create':
            return [AllowAny()]
        
        # Flujo de protección estándar: Criptografía válida + Contexto correcto
        return [IsAuthenticated(), IsTenantUser()]
```

## 6. DIAGNÓSTICO DE OPERACIONES (PRUEBAS EN POSTMAN)
Tras la ejecución de este módulo, las respuestas esperadas ante flujos de peticiones cruzadas deben responder a la siguiente matriz de control:

| Origen del Token | Endpoint Solicitado | Código HTTP Esperado | Estado del Flujo Técnico |
| :--- | :--- | :--- | :--- |
| Token Tenant Válido | `/api/tenant/usuarios/` | `200 OK` | Acceso Autorizado (Éxito) |
| Token Admin Válido | `/api/tenant/usuarios/` | `403 Forbidden` / `401` | Interceptado por Validación de Claim |
| Token Modificado/Falso | `/api/tenant/usuarios/` | `401 Unauthorized` | Rechazado por Firma Inválida |
