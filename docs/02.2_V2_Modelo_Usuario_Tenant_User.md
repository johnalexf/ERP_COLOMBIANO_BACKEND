# 🏢 Arquitectura de Seguridad: Puerta 2 (Dominio Inquilinos)

Documentación técnica para la implementación del modelo de autenticación de inquilinos (`TenantUser`) y el sistema de autorización local. Este dominio opera bajo un aislamiento estricto en la base de datos `tenant`, utilizando una arquitectura desacoplada para garantizar la integridad de los datos por entidad.

---

## 1. Definición del Modelo de Datos (Dominio Inquilino)
La estructura del modelo de usuario para inquilinos se diseña para permitir una alta escalabilidad y un control granular sobre los campos de identidad.

**Archivo:** `apps/tenant/tenant_users/models.py`

### Justificación Técnica: AbstractBaseUser
A diferencia del dominio administrativo, el dominio de inquilinos implementa **`AbstractBaseUser`** por las siguientes razones técnicas:

* **Control de Esquema de Identidad**: Proporciona una implementación mínima de autenticación, permitiendo definir el correo electrónico (`email`) como el identificador único principal (`USERNAME_FIELD`).
* **Optimización de Recursos**: Elimina campos predefinidos innecesarios presentes en `AbstractUser`, reduciendo la carga en la base de datos `tenant`.
* **Seguridad por Segmentación**: Al no compartir la estructura del modelo administrativo, se mitiga el riesgo de escalada de privilegios hacia el panel de control central del ERP.

### Atributos Heredados de AbstractBaseUser
A diferencia de `AbstractUser`, que provee una estructura predefinida completa, la herencia de `AbstractBaseUser` otorga únicamente el núcleo de autenticación esencial de Django. El modelo `TenantUser` adquiere exclusivamente los siguientes elementos de forma nativa:

* **password**: Campo de longitud fija encargado de almacenar la representación encriptada de la clave mediante algoritmos de hashing.
* **last_login**: Atributo de tipo datetime que registra automáticamente la marca temporal del último acceso exitoso al sistema.
* **Lógica de Autenticación**: Provee los métodos internos necesarios para la verificación de credenciales y la gestión del estado de autenticación (ej: `is_authenticated`, `check_password`).

Cualquier otro campo adicional, incluyendo `is_active` o `date_joined`, debe ser declarado explícitamente en el modelo para que el sistema de base de datos `tenant` lo reconozca.

### Control de Acceso Basado en Roles (RBAC Local)
El sistema implementa una arquitectura **RBAC (Role-Based Access Control)** personalizada para el dominio de inquilinos. Este modelo permite la gestión autónoma de permisos dentro de cada base de datos de cliente sin dependencias del núcleo administrativo.

* **Aislamiento de Roles**: La definición de roles es local para la base de datos `tenant`. Las políticas de acceso de una entidad no tienen visibilidad ni efecto sobre otras entidades del sistema.
* **TenantPermission**: Entidad responsable de almacenar los códigos internos de permisos atómicos (ej: `view_invoice`, `create_user`).
* **TenantRol**: Entidad de agrupación que vincula múltiples permisos a una etiqueta de rol específica.

```python
from django.db import models
from django.contrib.auth.models import AbstractBaseUser
from django.utils import timezone

class TenantPermission(models.Model):
    name = models.CharField(max_length=255, verbose_name='Nombre del permiso')
    codename = models.CharField(max_length=100, unique=True, verbose_name='Código Interno')

    class Meta:
        verbose_name = 'Permiso de Inquilino'
        verbose_name_plural = 'Permisos de Inquilinos'

class TenantRol(models.Model):
    name = models.CharField(max_length=100, unique=True, verbose_name='Nombre del Rol')
    permission = models.ManyToManyField(TenantPermission, blank=True)

class TenantUser(AbstractBaseUser):
    email = models.EmailField(unique=True, verbose_name='Correo Electrónico')
    first_name = models.CharField(max_length=50, verbose_name='Primer Nombre')
    last_name = models.CharField(max_length=50, verbose_name='Primer Apellido')
    is_active = models.BooleanField(default=True)
    date_joined = models.DateTimeField(default=timezone.now)

    USERNAME_FIELD = 'email'

    class Meta:
        verbose_name = 'Usuario Inquilino'
        verbose_name_plural = 'Usuarios Inquilinos'
```

# 3. Serialización de Datos (Traducción JSON)
La capa de serialización se encarga de transformar los objetos del modelo `TenantUser` en representaciones JSON compatibles con el Frontend y de validar la integridad de los datos recibidos mediante peticiones HTTP.

**Archivo:** `apps/tenant/tenant_users/serializers.py`

### Justificación Técnica: Gestión de Credenciales en AbstractBaseUser
Debido a que el modelo `TenantUser` hereda de `AbstractBaseUser`, es imperativo implementar una lógica de creación que asegure el procesamiento del algoritmo de hashing para la contraseña.

* **Seguridad de Campo**: Se define el atributo `password` con el parámetro `write_only=True` para garantizar que la clave encriptada nunca sea expuesta en las respuestas de la API (operaciones de lectura).
* **Validación de Identidad**: Al haber configurado `USERNAME_FIELD = 'email'` en el modelo, el serializador valida la unicidad y formato del correo electrónico como credencial principal.



```python
from rest_framework import serializers
from django.contrib.auth.password_validation import validate_password
from .models import TenantUser

class TenantUserSerializer(serializers.ModelSerializer):
    """
    Serializador de grado industrial para la gestión de usuarios de inquilinos.
    Implementa validaciones estrictas de seguridad y encriptación de credenciales.
    """
    
    # Configuración de seguridad para el campo de contraseña
    password = serializers.CharField(
        write_only=True, 
        required=True, 
        style={'input_type': 'password'}
    )

    class Meta:
        model = TenantUser
        # Definición de campos expuestos en la interfaz de la API
        fields = [
            'id', 
            'email', 
            'first_name', 
            'last_name', 
            'password', 
            'is_active', 
            'date_joined'
        ]

    def validate_password(self, value):
        """
        Invoca los validadores de seguridad configurados en el núcleo del sistema
        para asegurar la robustez de la contraseña del cliente.
        """
        validate_password(value)
        return value

    def create(self, validated_data):
        """
        Sobrescritura del método de persistencia para garantizar 
        la ejecución del método create_user() del ORM, asegurando 
        el hashing de la contraseña antes del guardado físico.
        """
        user = TenantUser.objects.create_user(**validated_data)
        return user
```

### Protocolo de Validación RBAC (Local)
Durante la fase de serialización, el sistema se prepara para la futura integración de roles. Al ser una arquitectura **RBAC Local**, las validaciones de permisos se ejecutarán consultando exclusivamente la base de datos `tenant`, evitando latencias por consultas cruzadas a la base de datos `default`.

## 4. Lógica de Control (ViewSets)
La vista coordina el flujo entre el serializador y el modelo, aplicando las reglas de negocio y los permisos de acceso definidos para la Puerta 2.

**Archivo:** `apps/tenant/tenant_users/views.py`

```python
from rest_framework import viewsets
from rest_framework.permissions import IsAuthenticated, AllowAny
from django.conf import settings
from .models import TenantUser
from .serializers import TenantUserSerializer

class TenantUserViewSet(viewsets.ModelViewSet):
    """
    Conjunto de vistas (ViewSet) para la gestión integral de usuarios inquilinos.
    Provee automáticamente los endpoints para Listar, Crear, Recuperar, 
    Actualizar y Eliminar registros en la base de datos 'tenant'.
    """
    queryset = TenantUser.objects.all()
    serializer_class = TenantUserSerializer

    def get_permissions(self):
        """
        Implementa lógica de permisos dinámica basada en el entorno 
        y el tipo de acción ejecutada.
        """
        # Permite el registro público (creación) o acceso total en modo desarrollo
        if settings.DEBUG or self.action == 'create':
            return [AllowAny()]
        
        # Exige autenticación JWT para el resto de operaciones en producción
        return [IsAuthenticated()]
```

## 5. Enrutamiento (API Endpoints)
El sistema de enrutamiento define los puntos de acceso específicos para el dominio de inquilinos. Mediante el uso de un enrutador automático, se garantiza la generación de rutas estandarizadas para las operaciones CRUD sobre la base de datos `tenant`.

**Archivo:** `apps/tenant/tenant_users/urls.py`

### Justificación Técnica: Aislamiento de Rutas
La separación de las URLs en un archivo local por aplicación permite un mantenimiento modular. El uso de `DefaultRouter` asegura que las terminaciones de ruta (ej: `/id/`) sigan el estándar RESTful.

```python
from django.urls import path, include
from rest_framework.routers import DefaultRouter
from rest_framework_simplejwt.views import (
    TokenObtainPairView,
    TokenRefreshView,
)
from .views import TenantUserViewSet

# Inicialización del generador de rutas automáticas
router = DefaultRouter()

# Registro del prefijo 'tenant-user' vinculado a la lógica de inquilinos
router.register(r'tenant-user', TenantUserViewSet, basename='tenant-user')

urlpatterns = [
    # Endpoints de autenticación JWT para usuarios de negocio
    path('auth/login/', TokenObtainPairView.as_view(), name='tenant_token_obtain_pair'),
    path('auth/refresh/', TokenRefreshView.as_view(), name='tenant_token_refresh'),
    
    # Inclusión de rutas CRUD generadas por el router
    path('', include(router.urls)),
]
```

### 7.1. Protocolo de Pruebas Extendido (Operaciones de Escritura)
El conjunto de vistas `TenantUserViewSet` expone los métodos HTTP necesarios para la gestión completa del ciclo de vida del usuario en la base de datos `tenant`.

| Acción | Método | URL | Respuesta |
| :--- | :--- | :--- | :--- |
| **Registrar** | `POST` | `/api/v1/tenant/tenant-user/` | `201 Created` |

* **Nota de Seguridad**: Al ejecutar esta acción, el `ErpDatabaseRouter` garantiza que la eliminación solo afecte a la tabla residente en la base de datos `tenant`, protegiendo la integridad de la base de datos `default`.

## 8. Población de Datos de Negocio (Seeders Tenant)
La validación de la arquitectura Multi-Tenant requiere la inserción de datos de prueba específicamente en la base de datos de inquilinos para verificar el correcto funcionamiento del enrutador.

**Archivo:** `seed_tenant_users.py`

```python
import os
import django

# Inicialización del entorno Django
os.environ.setdefault('DJANGO_SETTINGS_MODULE', 'core.settings')
django.setup()

from apps.tenant.tenant_users.models import TenantUser

def seed_tenant_data():
    usuarios_negocio = [
        {
            "email": "vendedor1@empresa.com",
            "first_name": "Carlos",
            "last_name": "Pérez"
        },
        {
            "email": "contador1@empresa.com",
            "first_name": "Marta",
            "last_name": "Gómez"
        }
    ]

    print("--- Persistencia en Base de Datos TENANT ---")

    for data in usuarios_negocio:
        # El sistema detecta mediante el router que la consulta va a 'tenant'
        if not TenantUser.objects.filter(email=data["email"]).exists():
            # create_user asegura la encriptación de la contraseña
            TenantUser.objects.create_user(**data, password="TenantPassword123!")
            print(f"✅ Usuario Inquilino creado: {data['email']}")
        else:
            print(f"⚠️ El usuario {data['email']} ya existe en el dominio tenant.")

if __name__ == "__main__":
    seed_tenant_data()
```

## 9. Verificación de Aislamiento (Dual-Database)
Para confirmar el éxito de la implementación de la Puerta 2, se deben realizar las siguientes validaciones técnicas:

1. **Persistencia**: Verificar mediante `pgAdmin 4` que los registros creados a través del endpoint `/api/v1/tenant/tenant-user/` residan exclusivamente en la base de datos `erp_tenant_db`.
2. **Autenticación**: Validar que el inicio de sesión en `/api/v1/tenant/auth/login/` acepte únicamente el `email` como credencial, devolviendo un token JWT firmado con la `SECRET_KEY` del sistema.
3. **Restricción de Acceso**: Comprobar que los usuarios creados en el dominio `tenant` no poseen facultades de acceso hacia el dominio `master`, devolviendo un error `403 Forbidden` o `401 Unauthorized` según corresponda.

## 10. Gestión de Acceso (Flujo JWT Tenant)

Para validar la seguridad en el dominio de inquilinos, se debe ejecutar el flujo de intercambio de llaves digitales utilizando las credenciales del modelo `TenantUser`.

### Paso 1: Obtención del Token (Login Cliente)
A diferencia de la Puerta 1, este endpoint valida la identidad utilizando exclusivamente el correo electrónico como identificador.

1. **Método:** `POST`
2. **URL:** `http://127.0.0.1:8000/api/v1/tenant/auth/login/`
3. **Cuerpo (JSON):**
   ```json
   {
       "email": "vendedor1@empresa.com",
       "password": "TenantPassword123!"
   }
   ```
4. **Respuesta Exitosa:** El servidor retorna los objetos `access` y `refresh`. El token de acceso debe ser utilizado para las peticiones posteriores mediante la cabecera `Authorization: Bearer <token>`.

### Paso 2: Verificación de Restricción Cruzada (Seguridad Estricta)
Esta prueba confirma que el aislamiento de bases de datos impide que usuarios de negocio accedan a la administración central.

1. Obtener un token válido desde el endpoint de inquilinos (`/api/v1/tenant/auth/login/`).
2. Intentar una petición `GET` hacia un endpoint maestro: `/api/v1/master/admin-user/`.
3. **Resultado Esperado:** Código de estado `401 Unauthorized` o `403 Forbidden`.
   * **Justificación:** El `ErpDatabaseRouter` no encontrará el registro del usuario en la base de datos `default`, y la clase de permisos `IsAuthenticated` del ViewSet administrativo denegará el acceso al no localizar una identidad válida en su dominio de persistencia.

## 11. Resolución de Problemas (Troubleshooting)

Se documentan los errores más frecuentes durante la implementación del dominio de inquilinos y sus soluciones técnicas.

### A. Error: "Relation does not exist"
* **Causa**: Se intentó acceder a una tabla de inquilinos en la base de datos `default` o se omitió el flag de migración dirigida.
* **Solución**: Verificar la existencia de las tablas en `erp_tenant_db` y asegurar el uso del parámetro `--database=tenant` en el comando `migrate`.

### B. Error: "ALERTA ARQUITECTÓNICA" (ValueError)
* **Causa**: El `ErpDatabaseRouter` interceptó un intento de acceso de una aplicación no declarada en las listas `TENANT_APPS` o `ADMIN_APPS`.
* **Solución**: Registrar el `label` de la aplicación en el archivo `core/routers.py` según corresponda a su dominio de datos.

### C. Error: "Forbidden (403)" con Token Válido
* **Causa**: El token fue generado para un usuario administrativo (Puerta 1) e intentó ser utilizado en un endpoint de inquilinos (Puerta 2).
* **Solución**: Los tokens no son intercambiables entre dominios debido al aislamiento de modelos (`AdminUser` vs `TenantUser`). Se debe realizar una autenticación nueva en el endpoint `/api/v1/tenant/auth/login/`.

### D. Integridad Referencial: Bloqueo de Relaciones
* **Causa**: Intento de creación de un campo `ForeignKey` entre un modelo de la base de datos `default` y uno de `tenant`.
* **Solución**: Por directriz del método `allow_relation` en el enrutador, estas relaciones físicas están prohibidas para mantener el desacoplamiento total. Se debe utilizar una referencia lógica (almacenamiento del ID) en lugar de un vínculo de integridad de base de datos.
