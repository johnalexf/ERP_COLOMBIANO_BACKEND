# MANUAL TÉCNICO: COMPLEMENTO DE SEGURIDAD Y AISLAMIENTO MULTI-TENANT (DOMINIO ADMINISTRADOR)

## 1. OBJETIVO Y CONTEXTO ARQUITECTÓNICO
El propósito de este componente es establecer un aislamiento criptográfico y lógico para el entorno administrativo del sistema ERP. Se implementa una arquitectura simétrica a la del entorno inquilino (Tenant) para prevenir el cruce de credenciales.

Adicionalmente, se introduce un mecanismo de flexibilidad en la autenticación, permitiendo el ingreso al sistema utilizando tanto el correo electrónico (`email`) como el nombre de usuario (`username`), mediante el uso de consultas complejas en la base de datos.

## 2. CAPA 1: SERIALIZACIÓN Y AUTENTICACIÓN DUAL (SERIALIZERS.PY)
Se diseña un serializador dedicado exclusivamente al proceso de *login* administrativo. Este componente se encarga de la validación de credenciales antes de la emisión del token, implementando objetos `Q` de Django para habilitar la búsqueda bidireccional.

### Creación del `AdminLoginSerializer` en `apps/admin/admin_users/serializers.py`:
```python
from django.db.models import Q
from rest_framework import serializers
from .models import AdminUser

class AdminLoginSerializer(serializers.Serializer):
    """
    Serializador dedicado a la autenticación de Administradores.
    Permite el inicio de sesión mediante 'username' o 'email'.
    """
    username = serializers.CharField(required=True)
    password = serializers.CharField(
        write_only=True, 
        required=True, 
        style={"input_type": "password"}
    )

    def validate(self, data):
        identificador = data.get("username")
        password = data.get("password")

        if identificador and password:
            try:
                # Evaluación condicional (OR) sobre la tabla AdminUser
                user = AdminUser.objects.get(
                    Q(email=identificador) | Q(username=identificador)
                )
            except AdminUser.DoesNotExist:
                raise serializers.ValidationError("Credenciales inválidas o usuario no encontrado.")

            if not user.check_password(password):
                raise serializers.ValidationError("Credenciales inválidas.")

            if not user.is_active:
                raise serializers.ValidationError("Esta cuenta ha sido desactivada.")

            data["user"] = user
            return data
            
        else:
            raise serializers.ValidationError("Debe proporcionar un usuario/email y contraseña.")
```

## 3. CAPA 2: INYECCIÓN DE METADATOS EN EL PAYLOAD (VIEWS.PY)
Se estructura una vista basada en `APIView` para procesar la autenticación a través del serializador dual e inyectar la firma criptográfica correspondiente al entorno administrativo.

### Implementación de `AdminLoginView` en `apps/admin/admin_users/views.py`:
```python
from rest_framework.views import APIView
from rest_framework.response import Response
from rest_framework import status
from rest_framework_simplejwt.tokens import RefreshToken
from .serializers import AdminLoginSerializer

class AdminLoginView(APIView):
    """
    Procesa la autenticación de administradores y emite tokens estructurados.
    """
    serializer_class = AdminLoginSerializer

    def post(self, request):
        serializer = self.serializer_class(data=request.data)
        serializer.is_valid(raise_exception=True)
        user = serializer.validated_data['user']
        
        refresh = RefreshToken.for_user(user)
        
        # INYECCIÓN DEL CLAIM CRIPTOGRÁFICO
        refresh['user_type'] = 'admin'
        
        return Response({
            'refresh': str(refresh),
            'access': str(refresh.access_token),
            'message': 'Autenticación exitosa.'
        }, status=status.HTTP_200_OK)
```

## 4. CAPA 3: MOTOR DE AUTENTICACIÓN PERSONALIZADO (AUTHENTICATION.PY)
Mecanismo de resolución de identidad estricto que intercepta la validación del token para redirigir la consulta de persistencia de datos exclusivamente hacia el modelo `AdminUser`.

### Creación del archivo `apps/admin/admin_users/authentication.py`:
```python
from rest_framework_simplejwt.authentication import JWTAuthentication
from rest_framework_simplejwt.exceptions import AuthenticationFailed
from .models import AdminUser

class AdminJWTAuthentication(JWTAuthentication):
    """
    Clase de autenticación especializada para la aplicación Admin.
    Desvía la búsqueda de validación hacia la tabla AdminUser.
    """
    def get_user(self, validated_token):
        user_id = validated_token.get('user_id')
        user_type = validated_token.get('user_type')

        # Bloqueo inmediato ante discrepancia de origen de token
        if user_type != 'admin':
            raise AuthenticationFailed(
                'Token inválido para el entorno administrativo.', 
                code='invalid_token'
            )

        try:
            user = AdminUser.objects.get(id=user_id)
        except AdminUser.DoesNotExist:
            raise AuthenticationFailed(
                'Usuario administrador no encontrado.', 
                code='user_not_found'
            )

        if not user.is_active:
            raise AuthenticationFailed(
                'Esta cuenta administrativa está inactiva.', 
                code='user_inactive'
            )

        return user
```

## 5. CAPA 4: DIRECTIVA DE PERMISOS DE ACCESO CONTROLADO (PERMISSIONS.PY)
Filtro de seguridad a nivel de ruta que audita la reclamación inyectada en el JWT, garantizando el principio de menor privilegio.

### Creación del archivo `apps/admin/admin_users/permissions.py`:
```python
from rest_framework.permissions import BasePermission

class IsAdminUser(BasePermission):
    """
    Valida que el token portado contenga el claim 'user_type'
    con el valor 'admin' antes de autorizar el acceso a la vista.
    """
    def has_permission(self, request, view):
        if request.auth:
            user_type = request.auth.get('user_type', None)
            return user_type == 'admin'
            
        return False
```

## 6. INTEGRACIÓN Y CONFIGURACIÓN DE LA CAPA DE PRESENTACIÓN (VIEWS.PY)
Aplicación de las normativas de autenticación y permisos sobre el controlador principal de gestión de administradores.

### Modificación de `AdminUserViewSet` en `apps/admin/admin_users/views.py`:
```python
from rest_framework import viewsets
from rest_framework.permissions import AllowAny, IsAuthenticated
from django.conf import settings
from .models import AdminUser
from .serializers import AdminUserSerializer

# Importación de los componentes de seguridad
from .authentication import AdminJWTAuthentication
from .permissions import IsAdminUser

class AdminUserViewSet(viewsets.ModelViewSet):
    queryset = AdminUser.objects.all()
    serializer_class = AdminUserSerializer
    
    # SOBREESCRITURA DEL MOTOR DE AUTENTICACIÓN
    authentication_classes = [AdminJWTAuthentication]

    def get_permissions(self):
        # Entornos de desarrollo o endpoints de registro
        if settings.DEBUG or self.action == 'create':
            return [AllowAny()]
        
        # Bloqueo en producción: Token válido Y Perfil Administrador
        return [IsAuthenticated(), IsAdminUser()]
```
